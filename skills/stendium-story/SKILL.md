# История сборки инструмента → на Стендиум

Ты помогаешь автору превратить процесс сборки инструмента в пост-историю и опубликовать её на Стендиуме как **материал** (Resource), связанный ссылкой с самим инструментом. История — это отдельный портфолио-объект: «как это было сделано».

## Два железных правила

1. **Скраб перед публикацией.** История собирается из git-логов и заметок — туда легко попадают ключи, токены, пароли, .env-значения, email, телефоны. Их НУЖНО вырезать скрипт-скрабом **и** глазами. Публикуется только очищенный текст.
2. **Аппрув автора.** Показываешь автору финальный (уже очищенный) текст — он правит и говорит «публикуй». Без явного подтверждения не публикуешь.

## Предпосылки

- Токен `STENDIUM_TOKEN` в окружении (как в /stendium-publish; создать — `https://stendium.ru/profile/tokens`).
- Находишься в git-репозитории инструмента.
- Желательно знать ссылку на уже опубликованный инструмент (чтобы связать историю с ним).

## Шаг 1. Восстанови историю из репозитория

Собери факты (не выдумывай):
- `git log --oneline -50`, ключевые коммиты, `git shortlog -sn`, даты первого/последнего коммита (сколько заняло).
- `README`, крупные изменения, развилки. При наличии — сессионные заметки/чат сборки, если автор их даёт.
- Спроси автора о том, чего нет в логах: зачем делал, что было сложным, что бы сделал иначе.

Черновик истории (markdown), структура-ориентир:
- **Зачем** — какую задачу инструмент решает (1 абзац).
- **Как собирал** — ключевые решения, чем собрано, итерации.
- **Грабли** — что пошло не так, как чинил.
- **Итог** — сколько заняло, ссылка на инструмент (`https://stendium.ru/tool/<slug>`).

## Шаг 2. Скраб (гейт безопасности)

Запиши черновик в файл (напр. `/tmp/stendium-story.md`) и прогони скраб:

```bash
node <путь>/scripts/scrub.mjs /tmp/stendium-story.md
```

(путь к скрипту — рядом с этим SKILL.md, `scripts/`; при наличии `${CLAUDE_PLUGIN_ROOT}` это `${CLAUDE_PLUGIN_ROOT}/skills/stendium-story/scripts/scrub.mjs`.)

Скрипт создаст `*.scrubbed.md` и напечатает, что вырезал. **Разбери находки:**
- Если вырезаны **секреты** (ключи/токены/пароли) — это серьёзно: убедись, что в тексте их больше нет, и что автор понимает, что они были.
- **Сам** дополнительно вычитай текст на то, что regex мог пропустить (внутренние URL, имена клиентов, чужие ПДн, приватные детали).

## Шаг 3. Покажи автору и получи «да»

Покажи автору **очищенный** текст истории целиком (человекочитаемо) + список вырезанного. Дай поправить. **Дождись явного «публикуй».**

## Шаг 4. Опубликуй

Подготовь JSON истории:

```json
{
  "title": "Как я собрал <инструмент> (≤120)",
  "excerpt": "Тизер: о чём история, ≤280",
  "category": "engineering",
  "contentMd": "<очищенный markdown истории со ссылкой на инструмент>"
}
```

`category` — один из ключей тем (`product|growth|data|design|engineering|team|market|other`); для истории сборки обычно `engineering`. Запиши JSON в файл и опубликуй:

```bash
node <путь>/scripts/publish-story.mjs --story /tmp/stendium-story.json
```

Скрипт **ещё раз** скрабит перед отправкой и:
- при находках-**секретах** — ОТКАЖЕТСЯ публиковать (exit 3). Значит что-то просочилось: вернись к шагу 2, вычисти, покажи автору, повтори. Флаг `--force` — только если автор лично отсмотрел находки и подтвердил, что это ложные срабатывания.
- ПДн (email/телефон) — вырежет автоматически и предупредит; публикуется очищенная версия.
- при успехе — напечатает ссылку `https://stendium.ru/resource/<slug>`. Покажи её автору.

Токен скрипт читает сам из `STENDIUM_TOKEN` — не передавай его аргументом и не логируй. Секреты из кода/логов не публикуй ни при каких условиях.
