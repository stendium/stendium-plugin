# Публикация инструмента на Стендиум

Ты помогаешь автору опубликовать его рабочий инструмент на Стендиуме — витрине-портфолио, где инструмент находят и пробуют (люди и, в перспективе, ИИ-агенты). Твоя задача — собрать хорошую машиночитаемую карточку **из его проекта**, показать её автору, получить подтверждение и отправить.

## Железное правило

**Никогда не публикуй без явного подтверждения автора.** Сначала показываешь собранную карточку, автор правит и говорит «да» — только потом запускаешь скрипт. Публикация видна другим людям.

## Предпосылки: токен

Публикация требует персональный токен Стендиума в переменной окружения `STENDIUM_TOKEN`.

1. Проверь доступ: запусти бандл-скрипт с флагом `--check` (путь к скрипту — см. раздел «Запуск скрипта»).
2. Если токена нет — попроси автора:
   - Создать токен: `https://stendium.ru/profile/tokens` (кнопка «Создать токен»).
   - Задать в окружении: `export STENDIUM_TOKEN="stnd_..."` (bash/zsh) или `$env:STENDIUM_TOKEN = "stnd_..."` (PowerShell).
   - Токен показывается один раз — пусть сохранит.
3. Не проси автора вставлять токен в чат и не записывай его в файлы. Скрипт сам читает его из окружения.

## Шаг 1. Разберись, что за инструмент

Изучи текущий проект (не выдумывай — опирайся на факты):
- `README*`, `package.json`/`pyproject.toml`, `Glob`/`Grep` по коду — что это делает.
- Тип: веб-приложение, Telegram-бот, сайт, сниппет/скилл, API-эндпоинт.
- **Рабочая ссылка (`url`)** — где инструмент реально открывается (деплой, ngrok, t.me/бот, репозиторий). Если не можешь вывести — **спроси автора**. Без рабочей ссылки публиковать нельзя.

## Шаг 2. Собери карточку (черновик)

Заполни поля. Соблюдай лимиты (иначе сервер обрежет/отклонит):

- `title` — короткое название, **≤120** символов.
- `excerpt` — тизер для ленты (одно-два предложения, «зачем это»), **≤280**.
- `description` — подробнее (как пользоваться, для кого), **≤500**, опционально.
- `url` — рабочая ссылка (http/https).
- `category` — ровно один ключ:
  | ключ | о чём |
  |---|---|
  | `product` | продукт и стратегия, custdev, фреймворки, позиционирование |
  | `growth` | маркетинг и рост, привлечение, контент, GTM, реклама |
  | `data` | данные и аналитика, дашборды, метрики, юнит-экономика |
  | `design` | дизайн и UX, интерфейсы, прототипы |
  | `engineering` | автоматизация и разработка, скрипты, интеграции, боты |
  | `team` | команда и процессы, 1:1, ретро, апдейты |
  | `market` | тренды и рынок, обзоры, разбор кейсов |
  | `other` | всё остальное |
- `toolType` — ровно один:
  | тип | когда |
  |---|---|
  | `web_app` | открывается в браузере как приложение (будет встроено «потрогать» в iframe) |
  | `tg_bot` | Telegram-бот (укажи `machineCard.launchHint` = ссылка t.me/…?start=…) |
  | `website` | лендинг/сайт/статья |
  | `code_snippet` | сниппет/скрипт, который копируют |
  | `skill_plugin` | скилл/плагин для инструмента (напр. Claude Code) |
  | `api_endpoint` | вызываемый API |
  | `other` | не подходит ничего выше |
- `machineCard` — сердце находимости. Заполни:
  - `toolType` — то же значение, что выше.
  - `jobs` — **какие задачи инструмент решает, человеческими словами** («построить воронку по продукту», «собрать дайджест из чата»). Это главное — по этим формулировкам инструмент найдут. 2–5 штук.
  - `inputs` — что принимает на вход.
  - `outputs` — что отдаёт.
  - `techStack` — чем собрано (Claude Code, Next.js, n8n…).
  - `launchHint` — для `tg_bot`: deep-link `t.me/бот?start=…`; иначе `null`.
  - `embeddable`, `buildStoryUrl` — оставь `null` (сервер разберётся).
  - `authorContact` — @ник или ссылка для связи, опционально.

## Шаг 3. Покажи карточку автору

Выведи собранную карточку человекочитаемо (не JSON-портянкой). Спроси: всё ли верно, что поправить. Дай поправить `title/excerpt/description/jobs/category/toolType/url`. **Дождись явного «публикуй».**

## Шаг 4. Опубликуй

Запиши подтверждённую карточку в JSON-файл (временный, например в системную временную папку) строго в формате тела запроса:

```json
{
  "title": "...",
  "excerpt": "...",
  "description": "...",
  "url": "https://...",
  "category": "data",
  "toolType": "web_app",
  "machineCard": {
    "toolType": "web_app",
    "jobs": ["...", "..."],
    "inputs": ["..."],
    "outputs": ["..."],
    "techStack": ["..."],
    "embeddable": null,
    "launchHint": null,
    "buildStoryUrl": null,
    "authorContact": "@nick"
  }
}
```

Затем запусти скрипт (см. ниже). Разбери вывод:
- Успех → скрипт печатает ссылку `https://stendium.ru/tool/<slug>` и строку `{"ok":true,...}`. Покажи ссылку автору.
- **409** (уже опубликован инструмент с этим URL) → предложи автору обновить существующий: спроси `toolId` (из ссылки на инструмент) и перезапусти с `--update <toolId>`.
- Иная ошибка (400/401/429) → покажи автору текст ошибки от скрипта и помоги исправить (лимиты, токен, суточный лимит).

## Запуск скрипта

Скрипт лежит рядом с этим SKILL.md: `scripts/publish.mjs`. Собери его абсолютный путь от директории этого скилла (если доступна переменная `${CLAUDE_PLUGIN_ROOT}`, полный путь — `${CLAUDE_PLUGIN_ROOT}/skills/stendium-publish/scripts/publish.mjs`).

```bash
# проверить токен/доступ
node <путь>/scripts/publish.mjs --check

# опубликовать
node <путь>/scripts/publish.mjs --card /tmp/stendium-card.json

# обновить существующий
node <путь>/scripts/publish.mjs --card /tmp/stendium-card.json --update <toolId>
```

Токен скрипт читает сам из `STENDIUM_TOKEN` — не передавай его аргументом и не логируй.
